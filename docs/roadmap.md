# Roadmap · v1.1 / Phase 2 / ADR candidates
Документ агрегирует отложенные пункты из ревизии ADR (2026-05-31). Каждый пункт — открытый вопрос, сознательно отложенный из объёма эталонной версии v1.0 с явным обоснованием.
Исходный договорной объём v1.0 закреплён в [scope](scope.md) и [README](../README.md). Roadmap — аддитивный слой, не пересматривает решения v1.0.
## v1.1 — бэклог
Аддитивные расширения эталонной версии v1.0. Не ломающие совместимость изменения для потребителей; принимаются по мере возникающей потребности.
### Из [ADR-0001 · LOINC as primary identifier](adr/0001-loinc-as-primary-identifier.md)
- **Опциональное поле ****`display_localized`** в конверте успешного ответа. Эталонная версия v1.0 возвращает только английское длинное имя LOINC (независимое от языка). v1.1 добавляет необязательное локализованное отображение через расширение FHIR либо отдельное поле без ломающего совместимость изменения.
### Из [ADR-0002 · UCUM mandatory preprocessing + reject on failure](adr/0002-ucum-mandatory-preprocessing.md)
- **Таблица нормализации qualitative-значений** — аналог таблицы транслитерации UCUM для qualitative-ветки. Канонизация форм «отр.» / «negative» / «не обнаружено» / «отрицательно» / титров «1:80» в единое представление. В v1.0 — ответственность адаптера источника; v1.1 — таблица на стороне эталонной реализации с явным версионированием (аналогично артефакту транслитерации UCUM).
### Из [ADR-0006 · API contract + rejection envelope](adr/0006-api-contract-rejection-envelope.md)
- **Политика хранения отклонённых маппингов** (TTL против бессрочного). v1.0 хранит бессрочно (по умолчанию audit log); v1.1 фиксирует политику явно.
- **Локализация ****`error.message`** через машиночитаемый код + i18n-шаблон. Кандидат на отдельный ADR в v1.1 (см. бэклог ADR-кандидатов).
### Из [ADR-0007 · Method aliasing](adr/0007-method-aliasing.md)
- **Локализация поля LOINC COMPONENT** для UI-слоя (LOINC хранит на английском). Решается совместно с задачей локализации ADR-0001 (`display_localized`); возможно — единый ADR-кандидат «слой локализации на стороне эталонной реализации».
### Передача записей на ручную проверку
- **Проектная заметка** для сценария батча с частичными отказами. Когда из набора записей часть отклонена (конверт отклонения, см. [ADR-0006 · API contract + rejection envelope](adr/0006-api-contract-rejection-envelope.md)) или замаплена с низким confidence (альтернативы, см. [ADR-0003 · Explicit priority policy + alternatives in output](adr/0003-explicit-priority-policy.md)), такие записи образуют машиночитаемый набор «требует проверки». v1.1 фиксирует проектное описание этой передачи: какой минимальный контракт нужен потребителю, чтобы построить очередь на ручную проверку. Сама панель валидатора или врача — продукт приложения-потребителя, вне объёма эталонной версии (см. [scope](scope.md), явные исключения). Эталонная реализация обязана лишь сделать набор «требует проверки» программно идентифицируемым; v1.0 уже даёт это через конверт отклонения и confidence с альтернативами.
## Phase 2 — бэклог
Расширения за рамки лабораторных наблюдений эталонной версии v1.0. Структурные изменения, требующие отдельной major-разработки.
### Из [ADR-0004 · Semantic context mandatory](adr/0004-semantic-context-mandatory.md)
- **`patient_state`** (возраст, пол, беременность) как обязательное контекстное поле. v1.0 откладывает: последствия для PII + расширение поверхности контракта.
### Из [ADR-0005 · Version-aware mapping](adr/0005-version-aware-mapping.md)
- **Политика пакетного remap** — endpoint для пакетных операций (например, «remap всех маппингов с `current_version_status != current`»). v1.0 — только одиночный `/remap` (для одного маппинга).
- **Уведомление о релизе LOINC** — push-уведомления потребителям о выпуске новой версии и числе затронутых маппингов. v1.0 — pull через retrieval; Phase 2 — если push-уведомление потребуется, оно строится поверх эталонной реализации, а не внутри.
### Из [ADR-0006 · API contract + rejection envelope](adr/0006-api-contract-rejection-envelope.md)
- **Пакетные endpoints** (`POST /v1/map/batch`, `POST /v1/remap/batch`).
- **Потоковая / асинхронная обработка** для длительных кампаний `/remap`.
### Из [ADR-0007 · Method aliasing](adr/0007-method-aliasing.md)
- **`GET /v1/mappings?alias_group={group_id}`** — endpoint для межметодного retrieval. v1.0 — `alias_group` доступен в audit, фильтрация на стороне потребителя.
### Преобразование единиц с учётом вещества (mass↔molar)
- Из сокращения объёма v1.0: НЕ выполняется преобразование mass↔molar (требует поиска молярной массы вещества и таблицы коэффициентов). Кандидат на ADR-0009 в Phase 2 при появлении валидированной таблицы молярных масс для лабораторных аналитов.
## Бэклог ADR-кандидатов
ADR-кандидаты, идентифицированные при ревизии v1.0 и требующие отдельной разработки.
- **ADR-0009 · кандидат · Преобразование единиц с учётом вещества** — преобразование mass↔molar (Phase 2; см. секцию выше).
- **ADR-0010 · кандидат · Подключаемость резолвера + eval harness** — формализовать слой резолвера как заменяемый порт (длинное имя LOINC → код); фреймворк оценки с золотым набором данных `(raw_name_RU → expected_LOINC)`. Отложено до появления данных для оценки и готовой к продакшену реализации резолвера. Архитектурный фундамент уже зафиксирован: [architecture](architecture.md) описывает резолвер как абстрактный порт без привязки к конкретной модели.
- **ADR-кандидат · версионирование на уровне поведения** — как фиксировать семантические изменения API endpoint, не затрагивающие форму (без shape-bump). Из открытых вопросов ADR-0006. Триггер для разработки: первый зафиксированный случай семантического дрейфа в проде.
- **`error.message`**** · ADR-кандидат по локализации** — формализовать подход к i18n для ответов об ошибке (машиночитаемый код + локализованный шаблон, либо только машиночитаемый код с локализацией ниже по потоку). v1.1.
## Трекер валидации
Открытые вопросы, требующие реальных данных для разрешения. Закрываются по результатам прогона валидации (5 слоёв, \~10 часов)
### Из [ADR-0003 · Explicit priority policy + alternatives in output](adr/0003-explicit-priority-policy.md)
- **Порог для переопределения «правило против низкого score».** Нужен ли минимальный score у кандидата, чтобы приоритетное правило могло его выбрать, или правило всегда побеждает score? Решение v1.0: правило всегда побеждает (`rule_overrode_score=true` фиксируется в audit). Прогон валидации должен наблюдать реальные случаи и решить, нужен ли порог для переопределения (правило отклоняется, если score \< X, fallback на `score_based`).
