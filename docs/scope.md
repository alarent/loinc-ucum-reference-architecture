# Границы публикации (scope)
Документ фиксирует, что входит в каждую фазу публикации и что принципиально вне неё. Цель — защита от расползания: при появлении новой идеи проверяется по этому документу, а не добавляется по ситуации.
## v1.0 — архитектурный артефакт (текущий релиз)
Что входит:
- описание трёх слоёв: Входной адаптер, Ядро нормализации, Выходной интерфейс
- восемь архитектурных решений (см. ADR-индекс), каждое с компромиссами и альтернативами
- JSON Schema для входного и выходного контрактов (input.schema.json, output.schema.json, map-response.schema.json)
- 7 worked examples (00–06): входной формат и preconditions, happy path, контекстная чувствительность, конфликт маппингов, отказ при невозможности UCUM-нормализации, миграция между версиями стандартов, UCUM canonical conversion
- preconditions — требования к OCR / парсингу выше по потоку + граничные случаи и семантические инварианты, явно вне scope нормализации
- ucum-transliteration — правила и fixtures для стадии перед UCUM (локализованные единицы → UCUM ASCII)
- versioning — политика версионирования стандартов и API
- ip-statement — позиция о независимом IP + dual licensing decision (Apache-2.0 AND CC-BY-4.0)
- glossary — словарь терминов
- capability-levels — модель уровней зрелости внедрения (capability levels, ADR-0008): additive-only путь освоения слоя
Критерий готовности v1.0: внешний инженер за 60 минут чтения может пересказать архитектурные решения с обоснованиями.
## v1.1 — расширение reference (планируемое)
Что входит дополнительно:
- SNOMED CT как secondary layer для observations за пределами лабораторного домена (только design, не реализация)
- OMOP CDM и ATHENA compatibility — design слоя выгрузки в таблицу MEASUREMENT
- формализация конфигурации priority policy (declarative schema или DSL)
- детальный механизм миграции между версиями LOINC и UCUM с audit trail
- расширение output-контракта на microbiology results (другое пространство sample types)
- дополнительные 3–5 worked examples по обратной связи
Критерий готовности v1.1: enterprise-команда может адаптировать архитектуру под свой стек без обращения к автору.
## Phase 2 — за горизонтом reference (отдельный проект)
Что НЕ входит в reference-публикацию и оформляется как отдельный репозиторий или коммерческая имплементация:
- reference implementation на конкретном языке (Python или Go — решение откладывается до Phase 2)
- каталог адаптеров под реальные форматы: ru_lab_csv, HL7v2, FHIR Observation
- production-grade LOINC и UCUM data + таблицы миграций между версиями
- бенчмарки на репрезентативных синтетических датасетах
- deployment patterns (k8s, serverless)
- substance-aware conversion layer (mass↔molar через molecular weight lookup) — кандидат на ADR-0009 в Phase 2; в v1.0/1.1 reference-слой остаётся в UCUM-only домене
В README явно проговорить: Phase 2 — отдельный проект, не обещание расширения v1.x. Иначе аудитория начнёт ждать `pip install` и не получит.
## Что слой принципиально не решает (явные исключения из задачи)
- substance-aware unit conversion: mass↔molar для биомаркеров (например, креатинин mg/dL ↔ μmol/L) требует molecular weight lookup, что выходит за UCUM-only домен ADR-0002. Реф-слой в v1.0 работает исключительно в UCUM-домене: mass-LOINC и molar-LOINC — разные mapping_id, конверсия между ними — ответственность потребителя. Кандидат на Phase 2 (см. ниже)
- клиническую интерпретацию значений: reference ranges, критические пороги, клинические алгоритмы
- reference range policies — у них отдельный жизненный цикл и источники
- валидацию клинической корректности запроса — это ответственность приложения-потребителя
- OCR и парсинг исходных бланков — это задача выше по потоку (см. preconditions)
- разрешение конфликтов между разными версиями одного и того же стандарта на уровне семантики — версии живут параллельно, явная политика выбирает применимую (см. ADR-0005)
- интерфейс ручной проверки (панель валидатора или врача для записей, не прошедших автоматический маппинг): слой предоставляет лишь машиночитаемый сигнал — rejection envelope (см. ADR-0006) для отклонённых записей и confidence с альтернативами (см. ADR-0003) для сомнительных успешных, — а саму панель и процесс ручной проверки строит приложение-потребитель. Кандидат на проектную заметку в Roadmap (v1.1)
## Правило расширения scope
Любое добавление в scope v1.0 после публикации:
1. Открывается как issue с обоснованием
2. Если затрагивает архитектурный инвариант — формализуется как новый ADR или supersede существующего
3. Изменения в выходном контракте — major version bump API
4. Изменения, ломающие потребителей ниже по потоку — обсуждаются с явным миграционным гайдом
