# Versioning policy
Документ фиксирует, как версионируются три независимых измерения системы: версии стандартов (LOINC, UCUM), внутренние правила маппинга и публичный API.
## Три независимых оси версионирования
1. **Версии стандартов (выше по потоку).** LOINC обновляется дважды в год. UCUM пересматривается реже, с breaking changes в специфических граничных случаях. Эти версии диктуются внешними сообществами, мы их не контролируем.
2. **Версии правил маппинга (внутренние).** Конкретный набор LOINC-кандидатов, priority policy, решения по маппингу для конкретных источников данных. Версионируется по SemVer внутри слоя.
3. **Версии публичного API (output contract).** Структура output JSON, имена полей, обязательность. Версионируется по SemVer, breaking changes — новая major.
## Принципы
- API независим от стандартов. Версия LOINC может смениться без major-bump публичного API.
- Любой output несёт явную пометку, на какой версии LOINC и UCUM сделан маппинг.
- Audit trail сохраняет `policy_version` и `mapping_timestamp` — это даёт возможность диагностики «почему через два года вот этот тест стал интерпретироваться иначе».
## Решённые вопросы (закрыто ADR-0005)
- **Deprecated-коды LOINC.** Не переписываются молча: маппинг неизменяем, при чтении вычисляется `audit.current_version_status` (`deprecated`/`replaced`/`removed`) + `replaced_by`; явный re-mapping — по инициативе потребителя через `/remap`.
- **Lazy version-aware на чтении vs eager re-mapping.** Выбран lazy: версия маппинга зафиксирована в audit-snapshot, актуальность вычисляется в момент запроса; глобальный re-mapping не требуется.
- **Совместимость источников на разных версиях.** Версии живут параллельно; каждый mapping несёт свой snapshot (`loinc_version`/`ucum_version`), сопоставимость — на уровне канонических кодов и единиц.
## Открытые вопросы
- Версионирование transliteration-таблицы: отдельный SemVer vs вместе с UCUM snapshot (кандидат на патч ADR-0005)
- Behaviour-level versioning: семантические изменения, не меняющие shape контракта
