# ADR-0002 · UCUM mandatory preprocessing + reject on failure
## Метаданные
- **Номер:** 0002
- **Уровень внедрения:** L1 (qualitative-ветка — L3)
- **Статус:** Accepted (v1.0)
- **Дата:** 2026-05-29
- **Связанные ADR:** ADR-0001 (LOINC as primary identifier), ADR-0003 (Explicit priority policy), ADR-0005 (Version-aware mapping)
- **Связанные документы:** [scope](../scope.md), [preconditions](../preconditions.md)
## Контекст
Лабораторное измерение в исходных данных несёт два независимых семантических слоя: **что измерено** (аналит + sample type + method) и **в каких единицах**. LOINC покрывает первый слой, UCUM (Unified Code for Units of Measure) — второй. В реальных российских данных единицы приходят в произвольных форматах: `мкг/л`, `ug/L`, `мкг / л`, `mcg/L`, `µg/L`, иногда с опечатками и нестандартными разделителями.
Без нормализации единиц числовое значение измерения не имеет однозначной интерпретации. `15` без явной единицы не отличает норму от десятикратного превышения. При этом часть систем-источников действительно отдают только число — единица предполагается «по контексту» (по тесту, по лаборатории, по «здравому смыслу»). Это и есть антипаттерн, против которого направлено решение.
Ключевой архитектурный вопрос: **что делать, если единица не парсится в UCUM-каноническую форму** — отклонить запись, угадать по контексту, прокинуть «как есть» с пометкой?
Ограничения и силы, действующие на решение:
- потребители ниже по потоку (CDSS, аналитика, OMOP-выгрузка) рассчитывают на консистентность единиц; молчаливое «прокидывание сырого значения» приводит к клинически опасным ошибкам интерпретации
- угадывание по контексту (аналит → дефолтная единица) работает на 80% случаев и катастрофически ломается на оставшихся 20%, причём незаметно ниже по потоку
- в reference-архитектуре нет доступа к лабораторно-специфичному контексту; любая эвристика угадывания требует знаний, которых у нормализационного слоя нет и не должно быть
- audit trail и обратимость операций — фундаментальное требование медицинского домена
## Решение
**UCUM-нормализация единиц измерения — обязательный шаг pipeline. При невозможности привести единицу к канонической UCUM-форме запись отклоняется со структурированной ошибкой; молчаливое прокидывание или угадывание не допускается.**
Конкретика:
1. **Стадия pipeline.** UCUM-препроцессинг выполняется до LOINC-маппинга, не после. Причина: один и тот же аналит может иметь разные LOINC-коды в зависимости от единицы (массовая концентрация vs молярная), значит без нормализованной единицы маппинг не детерминирован.
2. **Канонизация.** Унифицируются регистр, пробелы, направление слеша, юникод-символы (`µ` → `u`, `мкг` → `ug` через таблицу транслитерации единиц). Результат — строка, валидная по UCUM-грамматике.
3. **Конверсия.** При различии между входной единицей и единицей выходного контракта выполняется численная конверсия с фиксированными коэффициентами (см. ADR-0006). Исходное значение и единица сохраняются в audit trail.
4. **Reject-политика.** Если строка не парсится UCUM-грамматикой или не существует пути конверсии к canonical unit для данного аналита, запись возвращается со статусом `rejected` и кодом ошибки (`UCUM_PARSE_FAILED`, `UCUM_NO_CONVERSION_PATH`, `UCUM_AMBIGUOUS_UNIT`). Числовое значение в выходной контракт не попадает.
5. **Никакого угадывания.** Запрещены эвристики «если аналит = ferritin, единица по умолчанию ng/mL». Подобные правила, если нужны, живут в адаптере источника (вне reference scope, см. preconditions).
6. **Qualitative observations минуют UCUM.** Если `raw_value` парсится как качественный результат (строки вида «отрицательно», «positive», «обнаружено», «не обнаружено», титры формата «1:80»), а не как numeric, UCUM-нормализация пропускается. Запись возвращается со статусом `success`: `value.numeric = null`, `value.string = <канонизированный качественный результат>`, `value.unit = null`, `value.ucum_canonical = null`. Парсинг numeric vs qualitative выполняется до UCUM-стадии Ядром нормализации (см. ADR-0006 секция «Два слоя отказа»). Reject-политика пункта 4 применяется только к численной ветке; для qualitative входной `raw_unit`, если присутствует, пишется в audit без UCUM-валидации.
## Альтернативы и почему отвергнуты
### А1. Молчаливое прокидывание сырой единицы
Прокидывать единицу «как пришла», без нормализации, оставляя интерпретацию ниже по потоку.
Отвергнуто: переносит проблему вниз по pipeline в форме, где её сложнее обнаружить. Каждый потребитель ниже по потоку вынужден решать ту же задачу самостоятельно, неконсистентно, без audit trail. Это и есть «свой абстрактный справочник», против которого построена вся архитектура.
### А2. Угадывание по контексту analyte
Использовать таблицу «аналит → дефолтная единица» для случаев, когда единица не распознана.
Отвергнуто по трём причинам: (а) reference-архитектура не должна содержать клинических знаний о дефолтах конкретных лабораторий; (б) дефолты различаются между регионами, лабораториями, временными периодами — централизованная таблица будет либо неполной, либо неверной для конкретного источника; (в) угадывание невозможно отличить от достоверного значения в выходном контракте без специальных полей, и эти поля будут проигнорированы половиной потребителей ниже по потоку.
### А3. Best-effort с confidence score
Пытаться нормализовать, при неудаче возвращать значение с пониженным confidence.
Отвергнуто: confidence в выходном контракте уже несёт другую семантику (см. open question по форме confidence) — степень уверенности LOINC-маппинга. Перегружать поле двумя независимыми источниками неопределённости делает его непригодным для фильтрации ниже по потоку. Кроме того, «значение есть, но недостоверное» — самая опасная категория данных в медицинском домене: оно проходит большинство фильтров и попадает в визуализацию.
### А4. Отложенная нормализация (lazy)
Хранить сырую единицу, нормализовывать по запросу потребителя.
Отвергнуто: нарушает принцип единственного места нормализации. Каждый потребитель будет делать это по-своему. Конверсионные ошибки станут распределённой, невоспроизводимой проблемой.
## Последствия
### Положительные
- Выходной контракт даёт строгую гарантию: либо запись имеет каноническую UCUM-единицу и корректное числовое значение, либо запись явно отклонена с причиной.
- Системы ниже по потоку упрощаются: им не нужно реализовывать собственную UCUM-логику, и они могут доверять единице без дополнительных проверок.
- Audit trail содержит исходную и каноническую единицу, что делает любую конверсию обратимой и проверяемой.
- Reject-политика делает пробел в данных видимым: источники, у которых много `UCUM_PARSE_FAILED`, попадают в фокус для улучшения парсинга выше по потоку. При молчаливом прокидывании этот сигнал теряется.
- Qualitative observations (серология, микробиология, скрининги) проходят normalization без искусственного reject из-за отсутствия UCUM-единицы; пример — Anti-HBs «отрицательно» в worked example 00. Output schema явно моделирует две ветки (quantitative и qualitative) через oneOf в `$defs/value`.
### Отрицательные и компромиссы
- Доля отклонённых записей на «грязных» источниках может быть существенной (по эмпирическим оценкам — единицы процентов на нормальных источниках, до 10–15% на нестандартизованных). Это требует явной коммуникации с владельцем источника и не маскируется.
- Адаптерам источников приходится либо нести более качественный парсинг единиц, либо принимать факт reject. Это смещает нагрузку «вверх по pipeline», что соответствует принципу разделения ответственности, но требует ресурсов на стороне адаптера.
- Транслитерационная таблица (`мкг` → `ug` и т. д.) требует поддержки и расширения. Версионируется как часть UCUM-конфигурации (см. ADR-0005).
- В граничных случаях (например, единицы для микробиологии или коагулограммы с нестандартной семантикой) может потребоваться расширение UCUM-словаря или явное исключение из задачи в scope.
### Нейтральные
- Производительность: UCUM-парсинг — O(длина строки), не является узким местом для лабораторных объёмов.
- API-контракт: статус `rejected` со структурированной причиной — часть выходного контракта v1.0 и не требует изменений при последующих ADR.
## Открытые вопросы
Все открытые вопросы закрыты на housekeeping pass 2026-05-31:
- **Дополнительный код ****`UCUM_OUT_OF_RANGE`****.** Решено: не добавлять в v1.0. Out-of-physical-range — клиническая интерпретация (диапазон допустимых значений для analyte), не UCUM-задача; ответственность потребителя. Базовый набор `UCUM_PARSE_FAILED` / `UCUM_NO_CONVERSION_PATH` / `UCUM_AMBIGUOUS_UNIT` остаётся каноническим.
- **Транслитерационная таблица.** Решено: живёт в reference как versioned-артефакт ([ucum-transliteration](../ucum-transliteration.md), часть UCUM-конфигурации с собственным версионированием, см. ADR-0005). Региональные расширения — ответственность адаптера источника.
- **UCUM для non-numeric.** Решено в Решение п.6 (qualitative ветка минует UCUM; success-envelope с `value.string`). Закрыто в patch'е 2026-05-30.
- **Канонизация qualitative-значений** («отр.» / «negative» / «не обнаружено» / титры). Мигрировано в v1.1 backlog ([Roadmap · v1.1 / Phase 2 / ADR candidates](../roadmap.md)) как «qualitative values normalization table». v1.0 — ответственность адаптера источника.
