# ADR-0004 · Semantic context mandatory
**Статус:** Accepted для v1.0
**Уровень внедрения:** L3
**Дата:** 2026-05-29
## Контекст
Лабораторные observations не существуют в вакууме. Один и тот же аналит с одним и тем же числовым значением может маппиться в разные LOINC-коды, иметь разную клиническую интерпретацию и требовать разной нормализации — в зависимости от семантического контекста.
Минимальный контекст, меняющий интерпретацию:
- **specimen** (sample type) — serum vs plasma vs whole blood → разные LOINC system axis
- **method** — HPLC vs immunoassay vs IFCC protocol → разные LOINC method axis (см. example 03)
- **fasting_status** — fasting vs random vs post-prandial → разные LOINC для глюкозы и липидного профиля
- **collection_datetime** — отложен в v1.1 (см. открытые вопросы). В v1.0 источник передаёт только дату на уровне запроса (`date`); более точное время забора кладётся в `raw_comment`, отдельного context-поля на уровне биомаркера нет
- **patient_state** (возраст, пол, беременность) — меняет reference ranges (Phase 2)
Стандартная ситуация в российском healthtech-ландшафте: источник передаёт `{name, value, unit}` и сырой комментарий, без структурированного контекста. Поэтому контекст **выводит ядро** — из `raw_comment` и типа материала (`type`), а не получает его готовым от источника. Потребитель не должен гадать или применять невидимые дефолты — иначе последствия маскируются и проявляются как клинические ошибки на горизонте месяцев.
Ключевая семантическая разница, которую обычно теряют: **«контекст известен» ≠ «контекст не выводится»**. Первое — конкретное значение; второе — `null` (ядро не смогло определить поле из сырых данных). Оба — валидные различимые состояния. «Отсутствующего поля» в этой модели не бывает: входной контракт context-полей не содержит.
## Решение
1. **Контекст — это выход ядра, а не поля входного контракта.** Ядро выводит фиксированный набор context-полей для v1.0: `specimen` (sample_type), `method`, `fasting_status` — из `raw_comment` и `type`. Входной контракт context-полей не содержит (см. input.schema.json). `collection_datetime` отложен в v1.1, `patient_state` — Phase 2.
2. **Каждое выведенное context-поле всегда присутствует в выходе, но значение может быть ****`null`****.** Если ядро не смогло определить поле из сырых данных — это `null` (не ошибка). Кода `CONTEXT_FIELD_MISSING` не существует: источник context-поля не передаёт, поэтому «пропустить» их нельзя.
3. **Три различимых состояния context-поля:** (а) конкретное значение (`"HPLC"`, `"fasting"`) — ядро определило поле; (б) `null` — ядро не смогло определить поле из сырых данных; (в) `"not_applicable"` — поле иррелевантно для этого аналита (например, `fasting_status` для ферритина). Четвёртого состояния «поле отсутствует» нет — поле всегда присутствует в выходе.
4. **Output транслирует context as-is**, включая `null` и `"not_applicable"` значения. Audit фиксирует:
	- `context_fields_null: [...]` — поля со значением `null` (реальный дефицит информации)
	- `context_fields_not_applicable: [...]` — поля со значением `"not_applicable"` (иррелевантно для analyte)
	- `context_completeness: "full" | "partial" | "minimal"` — рассчитывается **исключая** N/A-поля из знаменателя (иначе ферритин всегда был бы `partial` по `fasting_status`)
5. **Priority policy получает явный сигнал** «знаю / не знаю / неприменимо» и может принимать различимые решения. При `null` в критичных полях policy выбирает менее специфичный LOINC и помечает completeness="partial". `"not_applicable"` не уменьшает completeness — это эквивалентно «полю не требовалось быть заполненным».
## Альтернативы
### А1. Implicit omission разрешён, потребитель заполняет дефолты
Источник передаёт что хочет, отсутствующие поля заполняются дефолтами на нашей стороне (`specimen="serum"` для большинства маркеров и т.д.).
**Почему отвергнуто:** дефолты невидимы потребителю. Ошибки маскируются — клиницист не отличит «источник реально знает что serum» от «мы догадались, что serum». Это прямой путь к silent failures на длинном горизонте. Архитектурный принцип ADR-0001 («референсный слой принимает решения, не перекладывает на потребителя») применяется наоборот: здесь мы не должны принимать незаявленных решений за источник.
### А2. Open context-словарь, источник передаёт что хочет
`context: Record<string, unknown>` — любые поля, без фиксированной схемы. Priority policy читает что найдёт.
**Почему отвергнуто:** непредсказуемая поверхность. Priority policy нельзя писать декларативно (ADR-0003), JSON Schema не валидирует осмысленность. Каждый источник изобретает свой словарь — обратная сторона той же проблемы, которую решает LOINC для analyte-имён.
### А3. Контекст в отдельном вызове enrichment-API
Observation отдаётся минимальным `{name, value, unit}`, контекст подтягивается отдельным API-вызовом по `source.record_id`.
**Почему отвергнуто:** нарушает атомарность observation. Race conditions между основным вызовом и enrichment'ом. Удваивает поверхность интеграции. Контекст — часть наблюдения, не метаданные о нём.
## Последствия
### Положительные
- Контракт **самодокументируется** через JSON Schema — список context-полей виден без чтения кода.
- Audit показывает `context_completeness` — измеримая метрика качества интеграции источника. Можно строить SLO.
- Priority policy получает differentiated input — `method=null` и `method="HPLC"` ведут к различимым решениям с явным audit-следом.
- `null` как валидное значение **легализует частичную интеграцию**: источник может стартовать с минимальным контекстом и улучшать его инкрементально, без ломающих совместимость изменений в контракте.
### Отрицательные
- Источники с бедными метаданными дадут больше `null`-контекста (ядро не сможет вывести поля из скудных `raw_comment`/`type`), а значит ниже `context_completeness`. Это не ошибка интеграции, а измеримый сигнал качества — улучшается по мере обогащения сырых полей.
- JSON Schema жёстко закрепляет context-поверхность. Добавление поля = ломающее совместимость изменение для строгих validator'ов. Версионирование через ADR-0006.
- Дополнительная работа на стороне источника: явно различать «не знаю» (null) и «знаю что N/A» (например, fasting_status для не-глюкозного теста). Документация должна это объяснить однозначно.
## Открытые вопросы
Все открытые вопросы закрыты на housekeeping pass 2026-05-31:
- **Closed set context-полей vs extensible.** Решено: closed set для v1.0 (`specimen`, `method`, `fasting_status`) — обеспечивает декларативную priority policy и валидируемую JSON Schema. `collection_datetime` отложен в v1.1 (в v1.0 есть только дата на уровне запроса). Extensibility через `context_extensions: Record<string, unknown>` — Phase 2 если возникнет emergent need.
- **`patient_state`****.** Phase 2. Мигрировано в [Roadmap · v1.1 / Phase 2 / ADR candidates](../roadmap.md). v1.0 не включает из-за PII implications и расширения поверхности контракта.
- **Migration path** (warning-only mode duration, критерии перехода в enforce). Решено: deployment concern адоптера, не reference-decision. Reference layer предоставляет оба режима; адаптер решает, как переключать.
- **`context_completeness`**** enum vs процент.** Решено: фиксированный enum `["full", "partial", "minimal"]`. Читаемее для UX потребителя; три категории достаточно для prioritization/alerting, percent overkill — гранулярность не используется на практике.
- **Локализация context-значений** (`кровь` vs `blood`). Решено: English canonicals в reference (consistent с LOINC English invariant). Локализация — ответственность адаптера источника; reference не несёт mapping-таблицы для context-локализации.
