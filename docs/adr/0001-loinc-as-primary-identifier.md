# ADR-0001 · LOINC as primary identifier
## Метаданные
- **Номер:** 0001
- **Уровень внедрения:** L1
- **Статус:** Accepted (v1.0)
- **Дата:** 2026-05-29
- **Связанные ADR:** ADR-0002 (UCUM mandatory preprocessing), ADR-0003 (Explicit priority policy), ADR-0004 (Semantic context mandatory), ADR-0005 (Version-aware mapping)
- **Связанные документы:** [architecture](../architecture.md), [scope](../scope.md)
## Контекст
Нормализационный слой лабораторных данных должен отвечать на вопрос «что было измерено» в терминах единой кодовой системы. На выбор есть несколько кандидатов: LOINC (Logical Observation Identifiers Names and Codes), SNOMED CT (подмножество Observable Entity / Substance), внутренние проприетарные справочники лабораторий, национальные классификаторы.
Выбор первичного идентификатора — фундаментальное решение: он формирует ядро выходного контракта и определяет, как системы ниже по потоку (CDSS, аналитика, OMOP-выгрузка, FHIR-интероперабельность) связывают свои данные с нормализованными. Ошибка здесь — это переписывание половины архитектуры.
Ключевые силы, действующие на решение:
- домен v1.0 — лабораторные observations (химия, гематология, общие клинические анализы), не microbiology и не визуализационная диагностика
- требования федеральных инициатив (FHIR R4/R5, РФ — ЕГИСЗ, US — USCDI/Promoting Interoperability) явно называют LOINC как предпочтительный код для Observation.code
- OMOP CDM и ATHENA-vocabularies используют LOINC как исходный идентификатор для Measurement
- внутренние проприетарные справочники — типовой антипаттерн в российском healthtech: «мы мапим свой абстрактный справочник со всеми чужими», что формально работает, но делает каждую интеграцию отдельным проектом с N²-сложностью
- лицензирование: LOINC — бесплатен для любого использования, SNOMED CT требует affiliate license в странах-членах SNOMED International и платной лицензии иначе — критически важно для референсной публикации без барьеров к адопции
- гранулярность: LOINC кодирует измерение как 6-осевую комбинацию (component, property, time, system, scale, method), что точно соответствует природе лабораторного результата
**Замечание о SNOMED CT.** Вопрос «почему не SNOMED CT» возникает по инерции — в FHIR в целом SNOMED CT считается основной клинической терминологией. Для лабораторных observations это категориальная ошибка: LOINC и SNOMED Observable принадлежат разным классам инструментов. LOINC кодирует измерение как 6-осевую комбинацию; SNOMED Observable описывает клинический концепт наблюдения. Лабораторные observations канонично живут в LOINC. Для не-лабораторных доменов (procedures, anatomy) SNOMED CT — правильный выбор, но это вне scope reference v1.0.
## Решение
**LOINC — первичный идентификатор в выходном контракте для любого лабораторного observation. Другие кодовые системы (внутренние коды источника, национальные классификаторы) могут жить в выходном контракте как secondary identifiers, но не заменяют LOINC.**
Конкретика:
1. **Статус в контракте.** Объект `primary` (`{ system: "LOINC", code, display }`) — обязательный в любом успешном ответе (см. output.schema.json). Отсутствие LOINC-маппинга — сигнал reject-статуса (rejected-ветка), а не «nullable field».
2. **Альтернативы.** Поле `alternatives[]` — массив вторых/третьих LOINC-кандидатов с score'ами (см. ADR-0003). В v1.0 `alternatives[]` — **только LOINC**: каждый элемент имеет `system: "LOINC"` (const в output.schema.json), `code`, `display`, `score`. Не-LOINC идентификаторы источника в `alternatives[]` не попадают (см. п.4); это сознательно сужено относительно ранней редакции с discriminator-полем `coding_system` — uniform LOINC-массив проще для потребителя, а secondary-коды живут в audit/input_echo.
3. **Связь с FHIR.** Выходной контракт структурно мапится на FHIR Observation.code без промежуточных трансформаций: `primary.code` → Coding.code, `primary.system` = `"http://loinc.org"`, `primary.display` → Coding.display.
4. **Secondary identifiers.** Если источник принёс внутренний код (проприетарный справочник лаборатории, национальный классификатор), он сохраняется в audit trail / `input_echo` (исходные данные), но **не** в `alternatives[]` (там только LOINC, см. п.2). Самостоятельным «path of truth» он не становится; structured-поле под не-LOINC secondary identifiers — Phase 2.
5. **Display string.** В `primary.display` в контракте всегда возвращается LOINC long common name на английском — именно он служит human-readable подтверждением и сверению с внешними инструментами. Локализованные display — ответственность слоя ниже по потоку.
## Альтернативы и почему отвергнуты
### А1. Внутренний проприетарный справочник
Построить собственную таксономию и мапить в неё все внешние кодовые системы по мере необходимости.
Отвергнуто: это и есть антипаттерн, наблюдаемый в типовом ландшафте российского healthtech. Проблемы: (а) сложность интеграции растёт как N²: каждый новый партнёр — новый проект маппинга; (б) нисходящие системы не понимают коды без вашего маппинг-сервиса; (в) версионирование внутренней таксономии непрозрачно для внешнего мира; (г) при смене владельца или банкротстве компании данные теряют семантику. Reference-публикация именно этому противостоит.
### А2. Национальный классификатор (НКЛП РФ, ICD-10/11, и т. п.)
Использовать национальный классификатор лабораторных исследований как первичный.
Отвергнуто: (а) национальные классификаторы лабораторных исследований в РФ существуют фрагментарно (НКЛП покрывает процедуры, не observations; МКБ — диагнозы, не измерения); (б) референсная публикация с национальным первичным идентификатором не переносится между юрисдикциями; (в) ЕГИСЗ и российские FHIR-профили уже содержат LOINC как предпочтительный код для Observation.code.
### А3. Несколько «co-primary» кодовых систем
FHIR Observation.code допускает массив codings — выдавать LOINC и SNOMED равноправно, без структурного приоритета.
Отвергнуто: (а) потребителям ниже по потоку нужен детерминированный идентификатор для join'ов, агрегации, обучения моделей — «double truth» перекладывает выбор на каждого потребителя и воспроизводит фрагментацию; (б) версионирование двух систем с разными ритмами релизов удваивает сложность ADR-0005; (в) лицензионные ограничения SNOMED CT всё равно делают её недоступной для части адоптеров; (г) FHIR в IG-практике всё равно выбирает preferred coding — отложенный выбор превращается в имплицитный.
## Последствия
### Положительные
- **Однозначный path of truth.** Системы ниже по потоку работают с одним первичным ключом для join'ов, OMOP-выгрузок, FHIR-экспорта. Сложность интеграции из N² превращается в N (каждый партнёр интегрируется один раз с LOINC).
- **Совместимость с FHIR и OMOP «built-in».** Выходной контракт мапится на FHIR Observation.code и OMOP MEASUREMENT без промежуточных трансформаций.
- **Нулевые лицензионные барьеры.** LOINC бесплатен и для коммерческого использования; референсная публикация может быть взята enterprise-адоптером без procurement-блокеров.
- **Совпадение с регуляторными ожиданиями.** И РФ (ЕГИСЗ, профиль ИПС), и международные IG ожидают LOINC в Observation.code; референсная публикация идёт по течению, а не против.
- **Гранулярность в тему.** 6-осевая модель LOINC соответствует природе лабораторного результата и совместима с ADR-0004 (Semantic context mandatory).
### Отрицательные и компромиссы
- **Покрытие.** LOINC покрывает лабораторные observations хорошо, но не покрывает microbiology в полном объёме (это в v1.1) и не покрывает процедуры/анатомию (это SNOMED-территория, вне scope). Ограничение scope v1.0 явным образом (см. scope).
- **Русский язык.** LOINC имеет официальные переводы, но русский long common name покрывает не все коды и обновляется с задержкой. Reference возвращает английский display как «invariant truth»; локализация — ответственность слоя ниже по потоку.
- **Сложность resolver'а.** LOINC — большой справочник (\~95k кодов), поиск по нему нетривиален. Это создаёт необходимость в семантическом resolver (эмбеддинги типа BioLORD) и в ADR-0003 (priority policy) — они оба вытекают из выбора LOINC.
- **Версионирование.** LOINC выходит дважды в год; изменения в кодах и deprecations нужно обрабатывать в ADR-0005.
### Нейтральные
- LOINC parts (component, system, method и т. д.) не возвращаются в выходном контракте явно — они имплицитны в LOINC-коде и доступны через LOINC-базу; это сознательный выбор в пользу компактности контракта.
- LOINC Answer Lists (для качественных результатов) — отдельная тема, отложена в v1.1 или Phase 2.
## Открытые вопросы
Все четыре исходных открытых вопроса закрыты на housekeeping pass 2026-05-31:
- **Display long common name (всегда vs flag).** Решено в теле ADR (Решение п.5): английский LOINC long name возвращается всегда; локализация — ответственность Worked examples 01–03 валидируют.
- **`alternatives[]`**** + ****`coding_system`**** (uniform vs сегрегация).** Решено в теле ADR (Решение п.2): в v1.0 `alternatives[]` — только LOINC (`system: "LOINC"` const), discriminator-поле `coding_system` снято как избыточное; не-LOINC secondary — в audit/input_echo, structured-поле в Phase 2.
- **Deprecated LOINC коды.** Делегировано [ADR-0005 · Version-aware mapping](0005-version-aware-mapping.md): immutable snapshot + computed `current_version_status` + `replaced_by` + explicit `/remap` flow. Worked example 05 валидирует полный цикл.
- **Русский LOINC display в response.** Мигрировано в v1.1 backlog как опциональное поле `display_localized`. Reference v1.0 остаётся language-invariant (English LOINC long name); локализация решается через FHIR extension либо downstream UI layer без breaking change в v1.1.
