# Preconditions: требования к OCR / парсингу выше по потоку
Документ описывает, что нормализационный слой ожидает на входе. Сам OCR и парсинг бланков — явно вне scope данной reference-архитектуры. Здесь зафиксированы только требования к выходному формату системы-источника, чтобы дальше нормализация могла работать предсказуемо.
## Зачем разделение
Распознавание бланков и нормализация биомаркеров — две независимые задачи с разными паттернами отказа и разными командами компетенций. Их смешение приводит к тому, что ошибки OCR маскируются под ошибки нормализации, и наоборот. Чёткая граница на уровне структурированного JSON-контракта позволяет:
- развивать OCR-слой и нормализационный слой независимо
- тестировать каждый слой изолированно
- менять реализации OCR без затрагивания нормализационного ядра
- предъявлять конкретные требования команде выше по потоку, не претендуя на их экспертизу
## Требования к OCR / парсингу
1. **Корректная работа с десятичными разделителями.** Точка и запятая не должны теряться или подменяться. Значение `4,2` не должно становиться `42` или `4.2` без явной нормализации в каноническую форму.
2. **Распознавание клинически значимых знаков.** `>`, `<`, нестрогие неравенства, `±`, «плюс/минус», знак тире, маркеры «положительно/отрицательно» — обязательны к корректному распознаванию.
3. **Надёжное считывание цифр и единиц измерения.** Включая локализованные варианты: «мл», «г/л», «ммоль/л», «мкг/мл», проценты, степенные нотации вида `10^9/л`.
4. **Сохранение табличной структуры бланка.** Должно быть однозначно понятно, какая единица и какой референсный интервал относятся к какому показателю. Плоский текст без сохранения структуры недостаточен.
5. **Учёт заголовков, задающих контекст.** Тип исследования, биоматериал, дата забора, идентификатор лаборатории — атрибуты, обязательные к извлечению.
## Контракт выходного JSON
Формат, в котором нормализационный слой ожидает данные от источника:
```json
{
  "lab": "Helix",
  "date": "2026-04-02",
  "type": "кровь",
  "biomarkers": [
    {
      "raw_name": "Гемоглобин",
      "raw_value": "110",
      "raw_unit": "г/л",
      "raw_ref": "новорожденные 134-205, до 1 года 94-141, от 1 до 14 лет 110-150",
      "raw_comment": null
    },
    {
      "raw_name": "Общее количество лейкоцитов",
      "raw_value": "9.8",
      "raw_unit": "10^9/л",
      "raw_ref": "2-12",
      "raw_comment": null
    }
  ]
}
```
Поля `raw_*` сохраняются как есть — без попытки парсинга или интерпретации. Это сырьё для нормализационного слоя.
## Что источник НЕ обязан делать
Чтобы граница была чистой, источник явно не отвечает за:
- сопоставление `raw_name` с какими-либо стандартными кодами — это работа Резолвера LOINC
- нормализацию единиц измерения к UCUM — это работа Нормализатора UCUM
- интерпретацию `raw_ref` (референсных интервалов) — это вообще задача ниже по потоку и вне scope нашего слоя
- определение sample type или метода измерения по структуре бланка — это работа Детектора контекста внутри нормализационного ядра
## Семантические граничные случаи и инварианты входа
Этот раздел фиксирует инварианты, которые синтаксически проверяются [input.schema.json](../schemas/input.schema.json), и семантические правила сверх схемы. Нарушение любого из этих правил → top-level `preconditions: "failed"` envelope с кодом `PRECONDITION_FAILED` и `error.path` в формате JSONPath (см. [map-response.schema.json](../schemas/map-response.schema.json)).
### Пустые и «quasi-пустые» значения
- **`biomarkers: []`** — запрещено (`minItems: 1`). Пустой массив биомаркеров не имеет семантики в нормализационном домене; источник должен вообще не вызывать `/v1/map`, если нечему маппить
- **`raw_name: ""`** — запрещено (`minLength: 1`). Пустая строка не эквивалентна «нет имени» — это индикатор OCR-потери. null также недопустим (поле required как string)
- **`raw_value: ""`** — запрещено (`minLength: 1`). Пустая строка не различима от OCR-сбоя или реально пустого результата; источник должен явно определиться. Для реально пустого результата (биомаркер не измерялся) биомаркер не включается в массив
- **`raw_unit: ""`** — запрещено в v1.0 (явно null или non-empty string, без промежуточного состояния). null — явное утверждение источника что единиц нет (типично для qualitative); пустая строка — неоднозначна и не принимается
- **`raw_ref: ""`** и **`raw_comment: ""`** — разрешены (в отличие от raw_unit); audit-поля допускают пустоту как валидное состояние. Однако рекомендуется null вместо пустой строки для однозначности
### Дубли и повторы
- **Дубли ****`raw_name`**** в ****`biomarkers`** — разрешены в v1.0. Валидный сценарий: один и тот же аналит измерен разными методами (например, HbA1c через HPLC и immunoassay, или повторный забор в одном бланке). Каждый возвращается отдельным envelope; корреляция по порядку в input (envelope\[i\] ↔ biomarkers\[i\])
- **Полный дубль объекта biomarker** (все 5 полей идентичны) — разрешён схемой, но симптом бага источника. v1.0 не дедуплицирует (каждый получит свой mapping_id). Обнаружение и предупреждение — ответственность адаптера источника
### Типы и форматы
- **`raw_value`**** — всегда string, не number.** Парсинг в numeric vs qualitative — ответственность Ядра нормализации. Причина: валидные варианты включают `<0.5`, `>1000`, `отрицательно`, `1:80`, `5,4` (запятая как десятичный разделитель)
- **Десятичный разделитель.** Источник НЕ обязан нормализовать запятую в точку — Ядро нормализации принимает оба. Но в пределах одного биомаркера не должно быть смешения (`5,4.2` — неоднозначно, OCR-сигнал)
- **`date`**** — точность.** Только ISO-8601 date (`YYYY-MM-DD`), без времени. Time-sensitive observations (глюкоза постпрандиально, кортизол в конкретный час) — time-информация в v1.0 пишется в `raw_comment`; datetime-поле — v1.1 кандидат
- **`lab`**** — свободная строка в v1.0.** Нет реестра известных лабораторий. Конкретные значения рекомендуются в формате «лаборатория» или «парсер-версия» (примеры: `helix`, `invitro`, `ocr-v3.2.1`)
- **Биоматериал (****`type`****) — свободная форма.** Канонизация в sample_type enum — работа Детектора контекста. Источник передаёт исходную формулировку («сыворотка крови натощак», «blood serum», «кровь из вены»)
### Качественные результаты (qualitative)
- **Качественные валидны.** `raw_value` вида «отрицательно», «positive», «обнаружено», «не обнаружено», титры `1:80` — валидные входные значения. Мапятся по qualitative-ветке (см. ADR-0002 пункт 6)
- **`raw_unit`**** для qualitative.** Ожидается null. Если источник присылает unit для qualitative выхода — записывается в audit без UCUM-валидации; UCUM-reject не срабатывает
- **Нет обязательной канонизации формы.** «отрицательно» / «отр.» / «negative» / «не обнаружено» — все валидны. Ядро нормализации выберет canonical форму для `value.string` (таблица канонизации — открытый вопрос ADR-0002)
### Что сейчас НЕ проверяется
- Консистентность `date` с реальностью (будущая дата? дата до 1900?)
- Длина строк (нет maxLength) — защита от злоупотреблений на уровне transport layer
- Кириллица или латиница в raw_name — оба принимаются, резолвер решает при нормализации
## Обработка ошибок источника
Два разных класса отказов, оба возвращаются через top-level `preconditions: "failed"` envelope ([map-response.schema.json](../schemas/map-response.schema.json)):
- **JSON Schema валидация** (синтаксический слой) — невалидный JSON, отсутствие required полей, неверные типы, additionalProperties. Код ошибки: `INPUT_SCHEMA_VIOLATION`
- **Семантические invariants** (из этого раздела выше) — вход синтаксически валиден, но нарушает инвариант. Код ошибки: `PRECONDITION_FAILED`
Оба случая включают `error.path` в формате JSONPath до проблемного поля (например `$.biomarkers[2].raw_value`) и `audit` с версиями LOINC/UCUM в момент запроса. `mapping_id` не выдаётся для preconditions-отказа (решение ADR-0006).
Нормализационный слой не пытается «починить» битый upstream — это привело бы к маскировке OCR-багов.
