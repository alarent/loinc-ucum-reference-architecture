# Architecture (расширенная спецификация)
Этот документ — спина публикации. Расширяет README и раскрывает архитектурные слои, контракты, точки расширения, политику версионирования и региональный контекст.
## Системный разрыв в индустрии
Интероперабельность медицинских данных — системная боль глобального healthtech, особенно острая в российской экосистеме, где исторически каждая крупная платформа собирала собственные справочники. Для лабораторных данных задача формально решена: есть LOINC (Logical Observation Identifiers Names and Codes) и UCUM (Unified Code for Units of Measure) — открытые международные стандарты, покрывающие большую часть случаев.
Но типовые реализации собирают эти стандарты как абстрактный справочник без архитектурного слоя. Характерные паттерны отказа:
- **Код без единиц.** Тест сопоставлен с LOINC-кодом, единицы измерения лежат в отдельном строковом поле. `ng/ml`, `нг/мл`, `ng/mL`, «нанограмм на миллилитр» — четыре разные записи для системы.
- **Потеря семантического контекста.** Ферритин в сыворотке и ферритин в плазме при определённых протоколах не взаимозаменяемы. Типовой слой сохраняет `{code, value}` и теряет тип образца, метод, признак натощак.
- **Справочники без версионирования.** LOINC обновляется дважды в год, UCUM — с ломающими совместимость изменениями в граничных случаях. Маппинг, сделанный на LOINC@2.74, через два релиза начинает тихо деградировать.
- **Конфликты маппингов на уровне интеграции.** Один и тот же тест может иметь несколько валидных LOINC-кодов. Команды решают это по ситуации в коде интеграции, каждая изобретая свои эвристики.
Результат — «работающая» интеграция, которая ломается на граничных случаях, теряет семантику при передаче между системами, и её корректность принципиально непроверяема извне.
Проблема не в стандартах, а в отсутствии архитектурного слоя поверх них.
## Эмпирическое подтверждение
Архитектурные инварианты слоя сформулированы как структурные требования независимо от того, какими стандартами они реализуются. LOINC и UCUM выбраны как наиболее зрелая реализационная лексика, не как отправная точка дизайна.
Эмпирически разрыв подтверждается типовым ландшафтом российского healthtech: крупные federated-платформы строят интеграцию через внутренние «абстрактные справочники биомаркеров» без UCUM-канонизации и version-aware mapping. Это типовая, а не маргинальная картина.
## Региональный контекст (регулятивные предпосылки)
Российский healthtech-ландшафт движется в сторону HL7 FHIR: новые сервисы Минздрава строятся уже на нём, ЕГИСЗ и федеральные проекты подразумевают совместимость по FHIR Observation. Это меняет экономику интеграционного слоя:
- архитектура, изначально совместимая с FHIR Observation, открывает путь в ЕГИСЗ-совместимые продукты без переписывания на следующем шаге
- маппинг LOINC + UCUM — стандартная семантика для FHIR `Observation.code` и `valueQuantity`, не требует дополнительного слоя адаптации
- OMOP CDM / ATHENA — отдельный нисходящий канал для RWE и research-применений, не блокирует FHIR-совместимость
Этот раздел описывает региональный контекст как мотивацию, не как требование. Архитектура жизнеспособна и в юрисдикциях, где FHIR не является регулятивной нормой.
## Требования к нормализационному слою
Что слой обязан делать:
- резолвить локальные идентификаторы тестов в LOINC-коды с явным confidence и альтернативами
- канонизировать единицы измерения в UCUM на входе — без UCUM-формы значение в слой не попадает
- сохранять и возвращать семантический контекст (sample type, метод, темпоральные атрибуты) как структурированные поля, не опционально
- работать с несколькими версиями стандартов одновременно, явно фиксируя, на какой версии сделан маппинг
- давать audit trail по каждому нормализованному значению
Что слой не решает — см. scope, раздел «что слой принципиально не решает».
## Архитектура: три слоя
### Входной адаптер
Принимает сырые данные в форматах источников (HL7, FHIR, proprietary CSV/JSON, локализованные RU-справочники лабораторий). Задачи:
- парсинг исходного формата
- базовая валидация структуры
- локализация свободных полей до канонической формы на английском (внутреннее представление)
- упаковка в нормализованный raw-object для передачи в ядро
Не делает: семантической нормализации, обращения к стандартам, разрешения конфликтов. Это адаптер формата, не нормализатор.
### Ядро нормализации
Применяет последовательность преобразований:
1. **Резолвер LOINC.** Поиск кандидатов LOINC-кодов по `raw_name` через индекс DisplayName/component/system/property/long_name + семантический поиск по доменно-обученным эмбеддингам (например, SapBERT-XLMR, CODER, BioLORD-class, GAT-over-UMLS; либо лексический baseline типа RELMA — выбор имплементационный, не архитектурный). Получение top-N кандидатов с confidence.
2. **Детектор контекста.** Извлечение sample type, метода измерения, темпоральных атрибутов из заголовков и контекста бланка.
3. **Транслитерация UCUM (стадия перед UCUM).** Локализованные формы `raw_unit` («г/л», «ммоль/л», «МЕ/мл», «×10⁹/л») преобразуются в UCUM-валидные ASCII-строки (`g/L`, `mmol/L`, `[iU]/mL`, `10*9/L`) по явной таблице (см. `docs/ucum-transliteration.md`). Промах транслитерации → отказ с `UCUM_AMBIGUOUS_UNIT` или `UCUM_PARSE_FAILED`. Этот шаг снимает с UCUM-парсера локализацию — он работает только с ASCII UCUM-токенами.
4. **Нормализатор UCUM.** Приведение UCUM-строки к канонической форме (`g/dL` → `g/L` с фактором, см. ADR-0002). Если приведение невозможно — запись не идёт дальше в numeric-ветке; для qualitative observations UCUM выключается (ADR-0002 пункт 6).
5. **Резолвер конфликтов.** Применение priority policy к кандидатам, формирование финальной гипотезы и alternatives.
### Выходной интерфейс
Единый контракт для всех потребителей — независимо от системы-источника. Версионируется отдельно от версий LOINC и UCUM (см. versioning).
## Ко��тракты между слоями
**Входной адаптер → Ядро нормализации:** нормализованный raw-object с обязательными полями `{test_identifier, value, unit_raw, metadata}`. Всё, что не приведено к этой форме, до нормализационного ядра не доходит.
**Ядро нормализации → Выходной интерфейс:** полный структурированный объект с обязательными полями `{loinc, value, context, alternatives, audit}`. Отсутствие любого обязательного поля — ошибка, не допустимый граничный случай.
**Выходной интерфейс → потребитель (приложение):** контракт версионируется отдельно от версий стандартов. Ломающие совместимость изменения в API — новая major-версия интерфейса.

## Архитектурные решения
Восемь значимых решений вынесены в отдельные ADR с альтернативами и компромиссами:
- ADR-0001: LOINC as primary identifier
- ADR-0002: UCUM mandatory preprocessing + reject on failure (+ qualitative bypass)
- ADR-0003: Explicit priority policy + alternatives in output
- ADR-0004: Semantic context mandatory (с null-полями)
- ADR-0005: Version-aware mapping as first-class
- ADR-0006: API contract + rejection envelope (versioned independently of standards)
- ADR-0007: Method aliasing (deterministic alias group)
- ADR-0008: Capability gating via nullable fields — schema всегда полная, поля nullable пока уровень не включён; основа модели capability levels
Полный список с патчами — в ADR-индексе.
## Точки расширения
- **Новые источники данных** — через новые Входные адаптеры, без изменения ядра. Адаптер изолирует специфику источника.
- **Кастомные маппинги под корпоративную специфику** — через конфигурацию priority policy, не через модификацию кода слоя.
- **Secondary layer (SNOMED CT, proprietary)** — подключается как параллельный резолвер, результаты включаются в alternatives output. Design — v1.1.
- **OMOP CDM / ATHENA выгрузка** — параллельный нисходящий канал, не модифицирует основной output-контракт. Design — v1.1.
- **Новые версии стандартов** — подключаются без ломающих совместимость изменений в API через механизм version-aware mapping (см. ADR-0005, versioning).
## Что эта архитектура делает возможным ниже по потоку
Когда базовый слой LOINC + UCUM собран корректно, становятся реализуемыми задачи, которые иначе тонут в коде по ситуации:
- **Расчётные биомаркеры** (HOMA-IR, eGFR, индекс атерогенности и т. п.) — детерминированные формулы поверх стандартизованных значений с известными UCUM-единицами
- **Тренды и агрегация** — все значения по одному LOINC-коду живут в одном UCUM-пространстве, не требуют построчной нормализации
- **RWE и research-применения** — выгрузка в OMOP-совместимое хранилище без переписывания семантики
- **FHIR-совместимость** — `Observation.code` = LOINC, `valueQuantity.code` = UCUM, прямой маппинг
- **Cross-validation между источниками** — два разных бланка одного пациента сопоставимы напрямую
## Открытые вопросы
Фиксируются в Session log по мере появления. Ключевые вопросы v1.0 закрыты (2026-06-01):
- [x] **Confidence shape.** Решено: контракт несёт только число `audit.primary_score` `[0, 1]`; категории `{high, medium, low}` в envelope не входят, а выводятся потребителем из числа по документированным порогам (например, `high ≥ 0.9`). См. ADR-0003 и Schemas · индекс.
- [x] **Формализация audit trail.** Решено: гибрид — фиксированное обязательное ядро audit плюс опциональный расширяемый bag `audit.extensions` (`additionalProperties: true`); ядро остаётся строгим. Поле, ставшее несущим для интероперабельности, переносится в ядро через ADR. См. ADR-0003 (п.7).
- [x] **Стратегия миграции при изменениях в LOINC.** Решено: lazy version-aware на чтении — `audit.current_version_status` вычисляется в момент запроса, derived-маппинги доступны через `/remap` (`remap_source`). Глобальный re-mapping не требуется: версия маппинга зафиксирована в audit, актуальность считается на чтении. См. ADR-0005.
